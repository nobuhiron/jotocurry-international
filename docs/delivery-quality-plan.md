# 単発納品向け 軽量品質チェック計画（CI/E2E除外）

## 1. 目的
- 単発納品の静的サイトに対して、過剰な自動化を避けつつ、再現可能な品質確認を行う。
- コード/機能面の納品判定を、ローカル手動ゲートとして定義する。

## 2. 合格基準
- 不合格条件はエラーのみ（警告・ヒントは記録）。
- `astro check` エラーなし。
- `astro build` 成功。
- 主要5ページ x 2言語の手動確認完了。
- 問い合わせフォームの必須エラー表示と成功表示の確認完了。

## 3. 実施内容

### 3.1 ローカルコマンドチェック
- `pnpm check:delivery` または `scripts/run_local_checks.sh`

### 3.2 手動機能チェック
- 対象ページ（ja/en）
  - `/`
  - `/brand`
  - `/franchise`
  - `/contact`
  - `/privacy`
- 問い合わせフォーム
  - 必須未入力時のエラー表示
  - 正常入力時の成功表示（実送信なしの安全運用）

### 3.3 手動Lighthouse（納品直前1回）
- 対象: `/ja/`, `/ja/contact`, `/en/`
- 推奨閾値
  - Performance >= 90
  - Accessibility >= 95
  - Best Practices >= 95
  - SEO >= 95

## 4. 記録方式
- `scripts/new_report.sh` で `docs/delivery-quality-report-YYYYMMDD.md` を生成。
- 実行結果、未解決事項、最終PASS/FAILを記録する。

## 5. スコープ外
- デザイン品質の評価
- 翻訳品質の評価
- CI/E2E自動化の導入

# 単発納品向け 納品前チェック運用（Astro + Cloudflare / 個人運用最適）

> 目的：**納品事故（リンク切れ・ビルド失敗・フォーム不具合・本番環境差分）を最小化**しつつ、過剰な自動化は避ける。
> 方針：**「必須ゲート（自動）」＋「事故りやすい箇所の手動検収」＋「証跡を残す」**。

---

## 1. 前提（この運用の対象）
- 対象：単発納品の静的サイト（Astro / Cloudflare Pages）
- 目的：納品可否の判断を **再現可能な手順**にする（属人化・気分運用を避ける）
- スコープ：**コード/機能/公開品質**（デザインの良し悪しは別）

---

## 2. 合格基準（PASS/FAIL）

### 2.1 FAIL（不合格）条件
1つでも該当したら **FAIL**（修正して再チェック）。
- `astro check` がエラー
- `astro build` が失敗
- 主要ページで **意図しない 404 / 画像欠け / 重大な崩れ** がある
- 言語切替（ja/en）で主要導線が壊れている
- 問い合わせフォームで
  - 必須未入力時にエラーが表示されない
  - 正常入力時に成功表示が出ない（※実送信は不要）
- 本番（Cloudflare）でのみ起きる不具合（パス/OGP/環境変数など）が未解決

### 2.2 WARN（警告：記録のみ）
- Lighthouse 閾値未達（※ただし極端な低下はFAIL扱い可）
- 軽微な文言・余白・微小な表示差

---

## 3. 実行手順（この順番でやる）

### 3.1 事前確認（作業の明確化：30秒）
- Goal（何を直す？）：
- Scope（触る範囲/触らない範囲）：
- Done（終わりの条件）：

> 納品前は「ついで修正」を混ぜない。思いついた改善は TODO に積む。

---

## 4. 必須ゲート（自動）

### 4.1 ローカルチェック（必須）
以下のいずれかを実行し、**両方PASS**。
- `pnpm preflight`（推奨）
- または `pnpm check:delivery`

期待する結果：
- `astro check` → エラーなし
- `astro build` → 成功

#### よくある詰まり
- 依存差分がある場合のみ `pnpm install` を実行
  - 目安：ブランチ切替後に `package.json` / `pnpm-lock.yaml` が変わっている

---

## 5. 必須ゲート（手動検収）

### 5.1 主要ページ確認（ja/en）
対象（各ページを ja/en で確認）：
- `/`
- `/brand`
- `/franchise`
- `/contact`
- `/privacy`

チェック観点：
- **ナビ/フッター**：主要導線が機能、言語切替で破綻しない
- **リンク**：内部リンクが 404 にならない（意図しない404なし）
- **画像**：欠け/崩れ/不自然な引き伸ばしなし
- **SP/PC**：SPで致命的崩れ（可読性、CTA押下）がない

### 5.2 404ページ
- 404ページを用意している場合：
  - 誘導導線（トップへ戻る等）がある
  - デザインが最低限整っている
- 用意していない場合：
  - Cloudflareの既定404でも良いが、ポートフォリオ価値は下がる（推奨：用意）

### 5.3 問い合わせフォーム（必須）
目的：**「壊れてない」ことの確認**（実送信は不要）。
- 必須未入力 → エラー表示が出る
- 正常入力 → 成功表示が出る
- 二重送信・ローディング状態が破綻しない（可能なら）

---

## 6. 本番相当チェック（Cloudflare / 必須）

### 6.1 プレビュー or 本番URLでの再確認
ローカルではOKでも本番で壊れやすい項目を重点確認：
- **パス/ルーティング**（`base`・末尾スラッシュ・言語パス）
- **OGP / favicon**（SNSでのカード表示、`/ogp.png` 等の参照）
- **robots.txt / sitemap**（ある場合）

> 「ローカルOK、本番NG」は納品事故の典型。必ず本番相当URLで確認。

---

## 7. Lighthouse（納品直前1回：推奨）
対象（本番相当URLで実施）：
- `/ja/`
- `/ja/contact`
- `/en/`

推奨閾値（目安）：
- Performance >= 90
- Accessibility >= 95
- Best Practices >= 95
- SEO >= 95

運用ルール：
- 大幅未達（例：Performance < 80、A11y < 90）は **FAIL** 扱い
- 軽微未達は **WARN** として記録

---

## 8. 記録（証跡を残す：必須）

### 8.1 レポート生成
- `scripts/new_report.sh` で `docs/delivery-quality-report-YYYYMMDD.md` を生成。

### 8.2 レポートに残す内容（最低限）
- 実行日時
- 実行コマンド結果（preflight / build）
- 手動検収の結果（ページ・フォーム・404）
- Lighthouse結果（スコア）
- 未解決事項（WARN/FAIL）
- 最終判定：PASS/FAIL

---

## 9. スコープ外（この運用では扱わない）
- デザイン品質の主観評価
- 翻訳品質の主観評価
- E2E/Visual regressionなどの重い自動化

---

## 10. 付録：最小チェックコマンド

```bash
# 必須：ローカルゲート
pnpm preflight

# 差分確認（意図しない変更混入防止）
git status
git diff

# ステージ後の差分確認
git diff --staged
```
